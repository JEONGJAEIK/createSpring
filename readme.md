-- 이벤트리스너 상당히 어렵다.
* 일반적으로 옵저버 패턴은 발행자가 많은 리스너를 구독하고 있다. 리스너가 발행자를 구독하는것처럼 보이지만 발행자 클래스안에는 구독하는 리스너리스트가 들어있어야 함
* 즉 이벤트가 담당 리스너들의 리스트를 가지고있어야하는데 그럼 이벤트마다 다른 리스너 클래스를 만들어야한다.
* 그것은 이벤트 클래스가 하는데 스프링의 이벤트 클래스는 과거에는 이벤트버블리셔를 상속해야했지만 지금은 필요없다.
* 또한 이벤트퍼블리셔의 종류구분없이 하나의 객체만 쓰면 스프링에서는 사용이 가능하다
* 그렇다면 이벤트퍼블리셔에서 특정이벤트와 특정리스너와의 연관관계를 가지고 있을 것이다.
* 이것은 이벤트리스너가 달린 클래스가 어떤 이벤트를 수신하는지 의존관계주입때 처리할 것이라 생각한다.
* 실제로 확인해보니 어댑터 패턴을 사용한다. 그리고 처음부터 의존관계 주입때 처리되지도 않고 빈 후처리기가 지난후에 처리되고 있다.
* 그리고 그 어댑터는 빈으로 저장되지도 않고 별도의 인프라 로직으로 위치한다.
* 또한 이벤트리스너에 ASYNC나 트랜잭셔널이 붙어있을 경우엔 빈 후처리기에서 별도처리된다.
* 메서드 호출시 어댑터로 가서 거기서 빈을 찾음 프록시면 프록시호출하고 아니면 일반 빈 호출
